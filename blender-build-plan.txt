## 0. CONTEXT & GOALS

Problem statement
   The repo (~/godwyn-boss-fight on mossad; github.com/aaronblatnoy/
   godwyn-boss-fight) currently contains ONLY lore/spec text
   (boss-fight.txt, SPEC.txt, README.md) and one commit (c0e1688). There
   is no 3D asset, no render, no Blender install. We need a repeatable,
   fully-headless Blender pipeline that produces a Godwyn Phase-1
   character sheet and 7 moveset pose renders, authored entirely as bpy
   scripts (written by claude-fable-5) and executed via
   `ssh mossad "blender --background --python <script>.py"`.

Success criteria (measurable)
   - `blender --version` returns 4.x on mossad; `--background` runs a
     trivial bpy script and exits 0.
   - Cycles renders on GPU: a test render logs "Using OptiX" (or CUDA)
     and both RTX 3060 Ti devices are enabled (verified via a device
     dump script). A 1080p test frame completes in < 60s.
   - A single .blend (models/godwyn_phase1.blend) opens headless and
     contains: base mesh, armor, robe, hair, longsword, all materials,
     an armature, lights, and named cameras.
   - renders/character/ contains >= 6 stills (front, 3/4-left,
     3/4-right, back, side, face close-up, sword detail) at >= 2K.
   - renders/moveset/ contains exactly 7 stills, one per moveset pose,
     correctly posed and framed, matching the SPEC descriptions.
   - Every render reads as Elden Ring dark-fantasy: void/black bg with a
     faint vertical golden light crack; Godwyn is the scene's light
     source (SPEC line 337).
   - All artifacts committed and pushed to origin/main; README shows
     preview thumbnails.

Non-goals (explicitly out of scope)
   - Godot import / .glb export (a FOLLOW-UP plan; this stops at renders
     + a saved .blend). We DO keep the model export-friendly (see INV-4).
   - Rigged/keyframed ANIMATION or video. We render STILL poses only.
   - Sculpted high-poly detail, normal-map baking, retopo, UV-unwrapped
     texture painting. Materials are procedural/PBR, not textured.
   - Phase-2 (death) Godwyn, corruption, crown, dark markings — SPEC
     line 296/304/310 explicitly forbid these for Phase 1.
   - Photoreal likeness of the fan art (Enzo Spag / @DOUJ, SPEC line
     295) — we synthesize the described silhouette, not copy art.

Assumptions made
   A1. Blender is available in Arch's `extra` repo as `blender` and
       ships with Cycles + the OptiX/CUDA kernels. If pacman's build
       lacks GPU kernels, fall back to the official tarball from
       blender.org (Phase 0 handles both branches).
   A2. The NVIDIA driver (610.43.02, already present) is sufficient for
       Cycles OptiX; the full CUDA toolkit (`nvcc`) is NOT required —
       Cycles ships its own kernels. Phase 0 verifies this empirically
       rather than installing the toolkit blindly.
   A3. mossad has enough free disk for Blender (~500MB) + renders
       (each 2K PNG ~5-15MB; ~15 renders => < 250MB). Verified in P0.
   A4. The gpu-quarantine.service noted in user memory removes a THIRD
       (MSI) card each boot; the 2 working 3060 Ti's are the render
       target. Plan assumes 2 usable CUDA/OptiX devices, not 3.
   A5. "Fable writes the bpy script" means: for each build phase,
       claude-fable-5 authors the .py locally, we scp it to
       mossad:~/godwyn-boss-fight/scripts/, run it headless, pull logs.
       The scripts are committed as the reproducible source of truth.
   A6. Scale: model at 3.2m, 1.4x human proportions (SPEC line 313).
       We build at real-world metric scale (Blender units = meters) and
       let the camera framing handle "scaled appropriately for render."
   A7. The SPEC's Color(r,g,b) values (lines 334-341) are Godot/linear
       values. We author Cycles materials to MATCH THAT INTENT in linear
       sRGB; exact node values may differ but the visual target is those
       colors. See Section 4.

## 1. ARCHITECTURE OVERVIEW

How this fits into the existing system
   This is the first ASSET-PRODUCING work in a currently text-only repo.
   It stands alongside the design docs and feeds the eventual Godot 4
   build. The pipeline is: Fable authors bpy → scp to mossad → run
   headless on GPU → pull logs/renders → commit. The .blend is the
   durable model artifact; the scripts are the reproducible recipe.

Affected surface (this is not a services codebase — map to dirs)
   - mossad host      : Blender install, GPU config, headless runtime
   - repo scripts/    : bpy Python (the "source code" of the model)
   - repo models/     : godwyn_phase1.blend (+ optional .glb later)
   - repo renders/    : PNG output (character/ and moveset/)
   - repo README.md   : preview embed
   - local workstation: where Fable authors scripts before scp

Data flow (ASCII)
   [Fable authors *.py] --scp--> [mossad:scripts/]
        |                              |
        |                     blender --background --python
        v                              v
   [git commit scripts]        [Cycles GPU render (2x 3060 Ti)]
                                       |
                          +-----------+-----------+
                          v                       v
                 [models/*.blend]          [renders/**/*.png]
                          |                       |
                          +----> git add/commit/push --> origin/main
                                          |
                                   [README preview embed]

Invariants this plan must preserve
   INV-1 (headless-only): NO GUI. Every Blender invocation is
         `blender --background --python <script>`. Claude Code CANNOT
         open interactive SSH — all remote calls are
         `ssh mossad "..."` one-shots (per root CLAUDE.md). Multi-line
         => `ssh mossad "cmd1 && cmd2"`.
   INV-2 (GPU-real): renders MUST use Cycles on GPU (OptiX preferred,
         CUDA fallback). A CPU-only render is a FAILURE, not a
         degraded pass — enforce cycles.device='GPU' AND assert at least
         one GPU device is enabled in each render script's preamble.
   INV-3 (spec-fidelity): colors, proportions, silhouette come from
         SPEC.txt lines 293-341, not from model imagination. Phase 1
         rules: no crown, no dark markings, barefoot, partial gold
         armor over exposed chest, blue robes, long partial-braid hair,
         luminous pale skin, low-hang-guard neutral. Any deviation is a
         bug.
   INV-4 (export-friendliness): even though .glb export is out of scope,
         build so a later export won't fight us: single character
         collection, apply-scale-friendly transforms, a clean armature
         with sane bone names, no ngons in deforming areas where
         avoidable, meaningful object names. Cheap now, priceless later.
   INV-5 (reproducibility): the .blend is regenerable from scripts.
         Scripts are deterministic (seed any randomness). The committed
         .blend is a convenience artifact, not the only source.
   INV-6 (idempotent scripts): each build script clears/rebuilds its
         own objects by name so re-running doesn't duplicate geometry
         (e.g. delete existing "Godwyn_Body" before recreating). Enables
         safe iteration.
   INV-7 (light-source-is-Godwyn): SPEC line 337 — Godwyn's emission
         casts light (radius 12m, Color(1.0,0.92,0.6)). The void is lit
         BY him plus the golden crack, not by a neutral studio rig that
         would kill the dark-fantasy mood.

### PHASE 0: Environment Setup on mossad — get headless Cycles/GPU live
--------------------------------------------------------------------
   Prerequisites: none (SSH to mossad already works).
   Deliverables:
      - Blender 4.x installed and on PATH on mossad
      - scripts/ models/ renders/{character,moveset} dirs in the repo
      - scripts/00_env_check.py proving GPU render works
      - a committed 1080p test render (throwaway) demonstrating GPU path
   Steps:
      1. `ssh mossad "sudo pacman -Sy --noconfirm blender"` (or, if that
         build lacks OptiX kernels or pacman is undesirable, download the
         official tarball:
         `ssh mossad "cd ~ && curl -LO <blender.org LTS tarball> && tar
          xf blender-*.tar.xz"` and symlink into ~/bin). Decide branch
         empirically: install, then run the device dump in step 4; if no
         GPU devices appear, switch to the tarball.
      2. `ssh mossad "blender --version"` — confirm 4.x, exit 0.
      3. Create dir structure + .gitkeep + .gitattributes:
         `ssh mossad "cd ~/godwyn-boss-fight && git pull && mkdir -p
          scripts models renders/character renders/moveset && ..."`.
      4. Fable authors scripts/00_env_check.py: it imports bpy, sets
         engine=CYCLES, calls
         cycles_prefs.get_devices(), sets compute_device_type to
         'OPTIX' (fallback 'CUDA'), enables every GPU device, sets
         scene.cycles.device='GPU', PRINTS the device table, then
         renders a 1-cube 1080p test frame to /tmp and asserts the log
         contains an enabled GPU. scp + run headless; capture stdout.
      5. Verify: stdout lists both "NVIDIA GeForce RTX 3060 Ti" devices
         as ENABLED under OptiX (or CUDA), test frame written, render
         time logged.
      6. Check free disk: `ssh mossad "df -h ~ && free -h"`.
      7. Commit scripts/00_env_check.py (+ optionally the throwaway test
         render, or delete it) and push.
   Validation:
      - `ssh mossad "blender --background --python
         ~/godwyn-boss-fight/scripts/00_env_check.py 2>&1"` prints a
        device table with >=1 GPU ENABLED and "Using OptiX"/"Using CUDA",
        and exits 0. Test PNG exists in /tmp with non-zero size.
   Risks / Edge cases:
      - pacman Blender lacks OptiX kernels -> fall back to blender.org
        tarball (has kernels). Mitigation baked into step 1/4 branch.
      - Only 1 GPU visible (gpu-quarantine + missing card, per memory)
        -> acceptable; assert >=1, warn if <2, proceed.
      - 8GB VRAM per card is tight for heavy scenes -> keep scenes lean,
        prefer OptiX (lower overhead), enable "Persistent Data" off to
        avoid VRAM bloat; cap samples (see P4).
      - `sudo pacman` may prompt for a password non-interactively ->
        if it hangs, use the tarball route which needs no sudo.
      - OptiX unsupported on this driver combo -> CUDA fallback path is
        first-class, not a downgrade failure.
   Subagent orchestration:
      - Execution shape: [SINGLE] — gates the entire plan; nothing
        parallel until Blender+GPU are proven.
      - REQUIRED subagent_type: general-purpose
      - REQUIRED model: sonnet (install + verify shell work; not
        frontier reasoning). Escalate to opus ONLY if the GPU path
        refuses to come up and needs deeper diagnosis.
      - Run in background: no (blocks all downstream phases; result
        needed inline).
      - Brief template:
        "Execute Phase 0 of ~/godwyn-boss-fight/blender-build-plan.txt.

## 12. EXECUTOR ORCHESTRATION GUIDE

### 12.1 Parallelism map
   Phase | Execution  | Siblings | Depends on | Model  | Subagent type
   ------|------------|----------|------------|--------|---------------
   P0    | SINGLE     | —        | none       | sonnet | general-purpose
   P1    | SEQUENTIAL | —        | P0         | sonnet | general-purpose
   P2    | SEQUENTIAL | —        | P1         | sonnet | general-purpose
   P3    | SEQUENTIAL | —        | P2         | sonnet | general-purpose
   P4    | SEQUENTIAL | —        | P3         | opus   | general-purpose
   P5    | SEQUENTIAL | (<=2)*   | P4         | sonnet | general-purpose
   P6    | SEQUENTIAL | (<=2)*   | P4         | sonnet | general-purpose
   P7    | SEQUENTIAL | —        | P5 + P6    | haiku  | general-purpose

   * P5 and P6 run CONCURRENTLY with each other (both gated only on P4).
     Within P5/P6, an optional at-most-2-sibling fan-out (one per GPU)
     is allowed but NOT recommended (boilerplate ~= work; open-once-loop
     is cheaper). Do NOT exceed 2 GPU-bound siblings — only 2 cards.

   DAG (non-linear only at the P5/P6 fork):
     P0 -> P1 -> P2 -> P3 -> P4 -+-> P5 -+-> P7
                                 +-> P6 -+
     No specialized subagents (refractor-pair / consensus-verification-
     duo / policy-reader) are warranted — this is asset generation, not
     a cross-referenced refactor or a correctness-critical proof. Do NOT
     invoke them.

### 12.2 Cross-phase work consolidation
   - P1->P4 all edit the SAME .blend (built up statefully). They MUST
     run strictly sequentially (state dependency), never parallel.
   - P2's four sub-assets all edit scripts/02_details.py -> single agent,
     no fan-out (Edit-collision + shared .blend).
   - P5 and P6 both OPEN models/godwyn_phase1.blend read-only per
     process. Preferred pattern: each opens the .blend ONCE and loops
     its cameras/poses in one Blender process (avoids repeated
     multi-second load). They write to DISJOINT dirs
     (renders/character vs renders/moveset) so they never collide.
   - lib_godwyn.py is read by every phase; it grows across P0-P3 then is
     stable. Treat it as append-mostly; P4+ should not need to change it.

### 12.3 Model-mix budget
   Target split across the 8 phases:
     0% fable / ~12% opus (P4 only) / ~75% sonnet (P0,P1,P2,P3,P5,P6) /
     ~12% haiku (P7).
   Rationale: this is procedural-Blender scripting with a fully-pinned
   visual spec — squarely sonnet work. P4 alone earns opus (it fixes the
   rig topology + render config + color management that every later
   render inherits; a wrong call cascades). P7 is mechanical git/README
   -> haiku. ZERO fable: no phase here is frontier-grade; posing a rig
   and matching pinned RGB values is not the frontier. Assigning fable
   anywhere would be waste.

### 12.4 Estimated wall-clock
   Best case (P5||P6 parallel): P0..P4 serial (the bulk) + max(P5,P6) +
     P7. Dominated by modeling iteration + render times, not agent
     overhead. Rough order: a few hours of iteration, minutes of render.
   Worst case (strictly serial): P5 then P6 add one extra render batch.
   Critical path: P0 -> P1 -> P2 -> P3 -> P4 -> P6 -> P7 (P6 is the
     heavier of the two render phases: 7 poses vs 7 sheet cams, but P6
     also requires per-pose rigging, so it's the critical branch).

### 12.5 Failure recovery policy
   - P0 (env): retry up to 2x; if GPU still won't come up, switch pacman
     ->tarball branch; if BOTH fail, PAUSE and escalate (driver/OptiX
     incompat is an environment problem, not a code fix).
   - P1/P2/P3 (modeling/materials): retry up to 2x with same subagent;
     these are visual — if the preview is "runs but looks wrong," DO NOT
     silently accept; surface the preview and let the user judge
     (visual correctness is not machine-checkable).
   - P4 (integration/opus): on failure, PAUSE and escalate — rig/render-
     config decisions may need user input (resolution, look direction).
   - P5/P6 (renders): retry up to 2x; a single bad camera/pose re-renders
     in isolation (don't redo the whole batch). If deformation is ugly at
     extreme poses, log a deferred item and continue with the others.
   - P7 (commit): NEVER auto-push. Commit locally on mossad, then PAUSE
     and escalate for the user's push confirmation.
   Default if unspecified: pause and escalate.
   Visual-judgment note: because success is largely VISUAL, every render-
   producing phase should surface its preview to the user rather than
   self-certify "looks correct."

     renders/wip/<NN>_*.png           (gitignored; final renders live in
