## 0. CONTEXT & GOALS

The main animation loop (lifecycle/pending/workflows/godwyn-anim-critique-loop.workflow.js)
builds a Godwyn moveset animation with Fable, gates it on OBJECTIVE metrics
(anim_metrics.py), and each round asks a critic what to fix. This plan builds that
critic as a NESTED CHILD WORKFLOW `godwyn-vlm-critic` and rewires the parent's P2.B
to invoke it via workflow('godwyn-vlm-critic', {...}).

THE CRITIC (settled, empirically validated — build EXACTLY this):
A FULL-VIDEO, describe-then-judge, adaptive-swarm motion critic.
  - METRICS are the objective gate. anim_metrics.py (parent plan §4) computes the
    objective faults (cape/robe explosion via vertex displacement, body jerk/
    contortion, foot-slide, sword-detach, smoothness) and DECIDES pass/fail
    (G-4: metrics win). The VLM NEVER renders a verdict. Metrics own ALL objective/
    geometric faults — the VLM never inspects geometry.
  - The WIP animation is rendered to ONE FULL VIDEO (mp4). That video is the ONLY
    visual input to the VLM. FULL VIDEO ONLY — no key-frames, no single frames, no
    contact-sheet grids, ever. Every VLM task describes a narrow ASPECT of the full
    video.
  - Sonnet AUTONOMOUSLY decides what to probe. Each wave it chooses narrow motion/
    feel ASPECTS to have described ("describe the cape's motion", "describe the speed
    and snap of the swing", "describe how the body's weight shifts", "describe the
    hair physics", "describe how the cape settles after the strike"), grounded on
    {intended move + fablePassNote + metric flags + prior descriptions}. Open prompts,
    never leading, always "describe literally; do NOT assume it is intentional."
  - Those describe-tasks run as a SWARM: fired CONCURRENTLY to MAX OUT GPU CAPACITY.
    Sonnet reads the descriptions, decides what MORE to probe based on the feedback,
    keeps probing in waves. Budget-bounded (converge OR max-waves OR max-total-tasks)
    so it always terminates.
  - Sonnet then writes Fable a DETAILED PROMPT: it synthesizes ALL the aspect-
    descriptions + the metrics into one direction telling Fable exactly what to fix
    next. That is returned to the parent, whose P2.C Fable fixer consumes it.

EMPIRICAL GROUNDING (PROVEN — do not re-test):
  - Describe-then-judge beats asking the VLM to judge. The VLM only DESCRIBES;
    Sonnet does ALL judging.
  - The VLM reads MOTION qualities accurately from FULL VIDEO: a narrow cape-motion
    describe discriminated a broken clip ("erratic, chaotic, whips and flings
    outward, turbulent, snapping") from a normal swing ("fluid, gentle, settles
    naturally"). That is exactly what metrics give only as numbers.
  - Metrics own the objective + geometric faults (incl. cape explosion) AND the
    pass/fail gate. The VLM's job is the qualitative motion/feel description; it
    never judges.

BACKEND — THROUGH THE llm-server GATEWAY (hard requirement):
Aaron's network-abstraction directive: everything goes through the llm-server
gateway (repo aaronblatnoy/llm-server, on Coolify, black-sky:8000) so backends stay
swappable across machines. TODAY the gateway only proxies Ollama and its /chat is
text-only. So this plan UPGRADES the gateway into a model->backend ROUTER: an
OpenAI-compatible endpoint that carries MULTIMODAL content (video_url) and routes by
model name to the right backend — Ollama (text) or the local vLLM Qwen2.5-VL video
server (black-sky:8001). Both Ollama and vLLM already speak OpenAI
/v1/chat/completions incl. video_url, so the gateway just maps model->backend URL and
forwards (streaming pass-through). The describe-swarm then hits the GATEWAY
(black-sky:8000) with {model: the VL model, messages with video_url}; the gateway
routes to vLLM. Existing text /chat clients keep working (backward-compatible).

### Success criteria (measurable)

4. The gateway (aaronblatnoy/llm-server) is upgraded to the model->backend router with

6. Any null/timeout/empty swarm degrades to metrics-only (vlmTrust:'blind') — never a

### The four roles
- METRICS (anim_metrics.py): objective ground-truth backbone + the pass/fail GATE
  (G-4: metrics win). Owns all geometric/objective faults; produces the flags the
  critic probes AROUND. In C0.
- SONNET: the adaptive investigator + judge. C2.a picks which motion/feel ASPECTS to
  probe (autonomously, grounded on move+fablePassNote+metric flags+prior descriptions);
  C2.c reads descriptions and decides what to probe next; C3 synthesizes ALL
  descriptions + metrics into the detailed Fable prompt. DEFERS to metrics as ground
  truth (never overturns a metric FAIL; uncorroborated VLM color -> low-confidence
  notes).
- THE DESCRIBE-SWARM (VLM via the gateway->vLLM): a pure PERCEPTION oracle. It only
  DESCRIBES narrow aspects of the FULL VIDEO. Never judges. Fired concurrently to
  saturate the GPU.
- FABLE: builder/fixer OUTSIDE the critic (parent P1/P2.C). Consumes the fixBrief;
  emits a fablePassNote that feeds the next critique.

### Invariants (preserve)
- INV-1 Headless black-sky only (ssh "cmd"; no -t/GUI). Author scripts local, scp, run
  headless. /tmp on the server, NEVER ~/Desktop.
- INV-2 EEVEE for the WIP video render (fast). Blender 5.2 (scene.frame_set; slotted
  actions — no action.fcurves).
- INV-3 Idempotent scripts (delete-by-name before recreate).
- INV-4 Never auto-push. Commit local + server after phases.
- INV-5 The critic is READ-ONLY on the animation (renders + reads; never edits the rig).
- INV-6 LOCAL no-egress: gateway + vLLM both on black-sky; the video never leaves the box.
- INV-7 stability: the 7B/Ollama text path is light; the vLLM 3B video path is the
  heavy one. Size swarm concurrency to what vLLM sustains for video (measure); a hung/
  errored describe ABSTAINS, never blocks. Watchdog (sp5100_tco + RuntimeWatchdogSec=30)
  installed.
- INV-8 SPEC-fidelity: motion vocabulary from SPEC §7; the author-aspects node grounds
  probes on the intended move, not invented ones.
- INV-NEST One-level nesting: the child MUST NOT call workflow(). grep = 0.
- G-4 Metrics win. pass = overallMetricsPass AND (blind/timeout critic ? true :
  no swarm-surfaced blocker corroborated by a metric). A blind/timeout/empty critic
  abstains; metrics alone decide.
- INV-GW Everything the critic sends to a model goes THROUGH the gateway (:8000), never
  a raw backend URL.

DAG: P1 -> (P2 || P3 || P4) -> P5 -> P6(fold) -> P7 -> P8.
  P2 (video render script), P3 (gateway upgrade), P4 (swarm runner) are INDEPENDENT and
  can fan out after P1. P5 (child body) needs P1-P4. P7 (parent edit) needs P1+P5. P8 last.
Model mix: opus x1 (P5 child control-flow); sonnet for all else; fable ZERO (Fable lives
  in the parent build/fix, outside this critic); refractor-pair for P7 (surgical parent edit).
Failure recovery: P1 hard-stop if nesting unsupported (escalate, don't inline). P2/P3/P4
  retry once then escalate. P5 is the spine — if null after retry, escalate. P7 via
  refractor-pair. Any child node null/timeout -> abstain to metrics-only (G-4), never cascade.
Serialization: the child runs ONE describe-swarm at a time (sequential waves); the parent
  loop is serial. The gateway upgrade (P3) touches an EXTERNAL repo — do it once, verify,
  before P5 depends on it.
