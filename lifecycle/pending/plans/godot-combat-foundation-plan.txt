## 0. CONTEXT & GOALS

PROBLEM STATEMENT
  We have a locked design (SPEC.txt) and lore (boss-fight.txt) for a
  two-phase Elden Ring boss fight, plus a game-ready rigged hero mesh
  (models/godwyn_game.glb). Nothing playable exists yet — there is no
  Godot project on disk and Godot is not yet installed on the render
  server. This plan builds ONLY the COMBAT FOUNDATION for Phase 1
  (incl. the 50% "soft phase" lightning layer): the player, camera,
  combat core, the boss framework that expresses the SPEC state-based
  moveset tree, HUD, the void arena, and a vertical-slice checkpoint.

SUCCESS CRITERIA (measurable)
  S1. Project opens in Godot 4.x on the render server with Forward+
      (Vulkan) confirmed active; headless import/export runs clean.
  S2. Player: WASD locomotion, sprint, mid-roll with REAL i-frames
      (hurtbox disabled on frames 4-16 per SPEC line 143), stamina
      gate, light+heavy attack with animation-keyed hitboxes,
      flask heal (5x, 180 HP, cancel after 0.3s), death→"YOU DIED".
  S3. Camera: third-person default (SPEC §4) + lock-on at 25m with
      Q/E target switch, break at 35m, dynamic pullback hook.
  S4. Combat core: Area3D hitbox/hurtbox on dedicated collision
      layers (NOT physics bodies, NOT Godot physics hit detection —
      CLAUDE.md invariant), damage + poise application, hitstop via
      Engine.time_scale=0.05, knockback, i-frame handling, boss
      stagger on poise-break.
  S5. Boss framework: a state machine (IDLE/TELEGRAPH/ACTIVE/RECOVERY/
      STUNNED/MEMORY_FRAG) + an AttackData resource library + an AI
      decision layer that (a) picks initiators by distance weights,
      (b) walks the SPEC combo tree with probabilistic chain
      extension, (c) honors the Back-to-Player state and the
      Overshoot Rule, (d) unlocks the Phase 1.5 lightning layer at
      50% HP. Hitbox windows are driven by AnimationPlayer method/
      call tracks — NOT hardcoded timers where an animation exists.
  S6. HUD: player HP/FP/stamina bars, flask counter, boss HP bar +
      Cinzel name card, "YOU DIED" screen — all per SPEC §14, exact
      colors/positions.
  S7. The void arena (void.tscn) per SPEC §5 as the combat testbed.
  S8. VERTICAL SLICE: player enters the void, locks onto a Godwyn
      running a SUBSET of the moveset (X-combo tree + one gap closer
      + The Pause + Dragon's Memory), fights with roll/stamina/
      attacks, staggers him, and can die OR win. This is the "is it
      fun and ER-faithful?" checkpoint.

NON-GOALS (explicitly out of scope for THIS plan)
  N1. Phase 2 / Miquella (boss_p2, will_system, all §11 attacks).

## 1. ARCHITECTURE OVERVIEW

HOW THIS FITS
  This is the first executable code in the repo. Everything to date is
  the Blender character pipeline (scripts/, models/, renders/). This
  plan stands up res:// (SPEC §2 layout) INSIDE the repo (e.g.
  `game/` subdir per CLAUDE.md "The Godot game itself (res://) will
  live in a subdirectory"). It consumes models/godwyn_game.glb and the
  tuning reference; it produces the playable Phase-1 slice that Phase 2
  and the transition (future plans) will extend.

AFFECTED "SERVICES" (this is a single native game, but by subsystem)
  - engine/project  : Godot project, input map, render settings, CI.
  - player          : CharacterBody3D controller + stats + combat FSM.
  - camera          : third-person rig + lock-on.
  - combat-core     : hitbox/hurtbox Areas, damage/poise, hitstop.
  - boss            : boss FSM + AttackData library + AI + moveset tree.
  - ui              : HUD, boss bar/name card, death screen.
  - world           : void.tscn arena.
  - autoload        : game_state / game_manager singletons, TUNABLES.

DATA FLOW (one combat exchange)

  Input ─▶ player_controller ─▶ player_combat FSM
                                     │ enters ATTACK state
                                     ▼
                          AnimationPlayer plays "light_attack"
                                     │ call-track @ active frame
                                     ▼
                          player Hitbox(Area3D).monitoring = true
                                     │ area_entered
                                     ▼
                     boss Hurtbox(Area3D) ─▶ HitResolver
                                     │ apply damage + poise
                                     ▼
              boss_stats.hp -=, boss_stats.poise -=  ─▶ Hitstop(0.06s)
                                     │                 (time_scale=0.05)
                                     ▼
                   poise<=0 ? ─▶ boss FSM ─▶ STUNNED(0.8s)
                                     │
                                     ▼
                            HUD.update(boss_hp)

  Boss offense mirrors this: boss AI ─▶ boss FSM ─▶ AttackData ─▶
  AnimationPlayer clip ─▶ boss Hitbox active window ─▶ player Hurtbox
  ─▶ (if not in i-frames) player takes damage/knockback.

INVARIANTS THIS PLAN MUST PRESERVE (from CLAUDE.md + SPEC)
  I1. ANIMATION-DRIVEN COMBAT, NOT PHYSICS. Hit detection is Area3D
      hitbox vs Area3D hurtbox on dedicated collision layers. NEVER
      use Godot's physics-body collision or move_and_collide for
      combat hits. (CLAUDE.md "What NOT to do".)
  I2. HITBOX WINDOWS ARE ANIMATION EVENTS. Where an AnimationPlayer
      clip exists, its call/method track toggles the hitbox — not a
      standalone Timer. Timers are only a fallback for greybox clips
      that have not been authored yet, and even then they mimic the
      same event contract.
  I3. i-FRAMES ARE REAL. The roll disables the player HURTBOX on the
      exact frame window (SPEC frames 4-16), via monitoring toggle on
      a collision layer — NOT invulnerability-by-flag guesswork.
  I4. OVERSHOOT RULE (CLAUDE.md #5). Any boss lunge/thrust/jump that
      carries him past the player enters BACK-TO-PLAYER state; no
      neutral reset. The FSM MUST encode this transition.
  I5. SPEC-FIDELITY. All timings, damage, poise, colors, sizes come
      from SPEC.txt / the reference doc. Do NOT invent numbers.
      Any needed-but-absent number is flagged, not fabricated.
  I6. PHASE-1 APPEARANCE. No crown, no dark markings, no deathroot,
      Phase-1 material only. (Relevant when the rig is imported —
      do not apply Phase-2 shaders.)
  I7. HEADLESS-ONLY on the server. Import, export, and any automated
      build run via `ssh <server> "godot --headless ..."`. No GUI on
      the server. Interactive editing happens on the local Mac; the
      server runs headless builds and the native game.
  I8. IDEMPOTENT AUTOMATION. Import/build scripts are re-runnable and
      never duplicate state (mirrors CLAUDE.md #6 for bpy).
  I9. NEVER AUTO-PUSH. Commit locally/on server per phase; push only
      on explicit user instruction (CLAUDE.md #7).

### PHASE 1: Combat Core Primitives — Goal: hitbox/hurtbox + damage/poise/hitstop/knockback, testable in isolation
-----------------------------------------------------------------------
   Prerequisites: Phase 0 (layers, tunables).
   Deliverables:
     - hitbox.gd, hurtbox.gd (Area3D-based, layer-driven, I1).
     - hit_resolver.gd (damage, poise, damage_type, knockback vector,
       hit-once-per-swing set, faction filter).
     - hitstop.gd (Engine.time_scale=0.05 for duration then restore,
       robust to overlapping hits — reentrant-safe).
     - damage_types.gd enum.
     - A throwaway test scene: two capsules, one swings an Area, the
       other loses HP and gets knocked back — proves the pipeline with
       NO player/boss code yet.
   Steps:
     1. Hitbox: Area3D on player_hitbox/boss_hitbox layer, exports
        {damage, poise_damage, damage_type, knockback, owner_faction,
        hitstop_duration}. `activate()/deactivate()` toggle monitoring.
        Tracks already-hit hurtboxes this activation (no double-hit).
     2. Hurtbox: Area3D on *_hurtbox layer, exports {stats_target}.
        On area_entered from an enemy-faction hitbox → call
        HitResolver.resolve(hitbox, self).
     3. HitResolver: apply damage + poise to target stats; compute
        knockback along hitbox->hurtbox vector; emit `hit_landed`
        signal (owner, victim, amount, poise) for hitstop/HUD/AI.
     4. Hitstop: on hit_landed, set Engine.time_scale to tunables
        value for duration (unscaled timer), then restore to 1.0.
        Guard against nested calls (track a counter, restore only
        when all clear). Player i-frame check happens in Hurtbox
        BEFORE resolve (skip if hurtbox monitoring disabled).
     5. Wire the test scene; verify numbers come from tunables.
   Validation:
     - Test scene: attacker capsule swing removes exactly `damage`
       HP from victim, applies poise, and knocks it back; a second
       overlap in the same swing does NOT double-hit.
     - Hitstop visibly freezes for ~hitstop_duration then resumes;
       two hits in quick succession don't strand time_scale at 0.05.
     - Disabling the victim hurtbox (simulated i-frame) => no damage.
   Risks / Edge cases:
     - time_scale affects timers → use `Timer` with process_mode
       ALWAYS or `get_tree().create_timer(t, false, false, true)`
       (ignore_time_scale) for the hitstop restore, else hitstop
       never ends. This is THE classic bug; call it out in code.
     - Area monitoring toggled same-frame can miss/duplicate → toggle
       in physics_process and clear the hit set on activate().
     - Knockback vs CharacterBody3D → knockback is applied as a
       velocity impulse the controller reads next frame, not a
       teleport.
   Subagent orchestration:
     - Execution shape: [SINGLE] — small, load-bearing, everything
       downstream calls it; must be correct before fan-out.
     - REQUIRED subagent_type: general-purpose
     - REQUIRED model: opus (the time_scale/i-frame/double-hit
       correctness core — subtle bugs here poison all combat feel).
     - Run in background: no (player + boss phases both depend on it).
     - Brief template:
        "Execute Phase 1 of <PLAN_FILE_PATH>. Build the Area3D
         hitbox/hurtbox + damage/poise/hitstop/knockback core. Read §1
         (invariants I1-I3) and Phase 1. Touch only
         game/scripts/systems/{hitbox,hurtbox,hit_resolver,hitstop,
         damage_types}.gd and a test scene under game/scenes/_tests/.
         NEVER use physics-body collision for hits (I1). Pull all
         numbers from tunables.gd. Validation: the two-capsule test
         proves single-hit damage+poise+knockback, working hitstop
         that always restores, and i-frame skip. Evidence: describe
         the test run + paste the hitstop restore logic. Commit,
         no push. Print one-line summary and exit."

### PHASE 2: Player Locomotion, Stamina & Roll i-Frames — Goal: it MOVES and DODGES like Elden Ring
-----------------------------------------------------------------------
   Prerequisites: Phase 0; Phase 1 (hurtbox to toggle for i-frames).
   Deliverables:
     - player.tscn (CharacterBody3D + hurtbox + a placeholder capsule
       + a facing debug arrow + a hand-socket Node3D for future
       weapon/hitbox).
     - player_stats.gd (HP 500, stamina 100, FP 80 display-only,
       recovery model per SPEC §3).
     - player_controller.gd (walk 3.5 / sprint 6.5 / rotate 720°/s,
       lock-on strafe, gravity/floor).
     - Roll: 3.5m over 0.65s, stamina 15, i-frame window on frames
       4-16 (SPEC 143) implemented as a hurtbox-disable window on a
       normalized roll timeline; input-relative or lateral-to-target.
     - player_combat.gd FSM skeleton: FREE / SPRINT / ROLL / (attack
       states arrive Phase 3) — one authoritative state machine.
   Steps:
     1. Controller: read input, produce desired velocity; apply
        rotation at 720°/s; sprint drains 5 stamina/s; block sprint
        when stamina==0. Camera-relative movement (needs Phase 3's
        camera but works with a temporary fixed cam until then).
     2. Stats: stamina recovery delay 0.8s after any action, then
        25/s; paused while sprint/roll/attack (SPEC §3).
     3. Roll as an FSM state with a normalized t in [0,1] over 0.65s.
        Map SPEC frame window 4-16 (of an assumed clip length) to a
        t-range and disable the hurtbox for that sub-window (I3).
        Roll consumes 15 stamina on entry; movement follows a
        3.5m displacement curve; direction = input dir, or lateral
        to lock-on when locked with no directional input.
     4. Encode i-frame window numbers in tunables (from reference
        doc when present; SPEC frames 4-16 as placeholder). Draw a
        debug overlay (hurtbox color flips during i-frames) for QA.

## 4. DATA MODEL CHANGES

No database. The "data model" is Godot Resources + collision layers.

  Resource: AttackData (attack_data.gd) — the core combat datum:
    id: StringName
    role: enum { INITIATOR, BRANCH, EXIT, LIGHTNING }
    telegraph_time: float
    active_windows: Array[Dictionary]  # {start_t,end_t,shape,xform}
                                        # t normalized to clip [0,1]
    recovery_time: float
    damage: float
    poise_damage: float
    damage_type: enum (damage_types.gd)
    knockback: float
    animation_clip: StringName
    enters_state: enum { FACING, BACK_TO_PLAYER, AIRBORNE, EXTENDED }
    exit_options: Array[Dictionary]  # {attack_id, weight, continue_p}
    overshoot_possible: bool
    camera_pull: bool
    one_time: bool                   # Dragon's Charge

  Resource: MovesetTree (godwyn_p1.tres) — states + exits graph.
  Resource: Tunables (tunables.gd) — ALL numbers, repointed to
            elden-ring-combat-reference.md when it lands.

  Collision layers (the "schema" that must not churn): see §2.

  RLS / encryption / queues: N/A (single-player local game, no PII,
  no network, no persistence beyond a run). No compliance surface.

## 5. API CONTRACT

No network API. The internal "contracts" that later phases/plans
depend on:

  - Collision-layer map (§2) — frozen in Phase 0.
  - Input action names (§2) — frozen in Phase 0.
  - HitResolver.hit_landed(owner, victim, damage, poise) signal.
  - boss_base signals: state_changed, attack_started, poise_broken,
    hp_changed, threshold_crossed(pct).
  - player_stats signals: hp_changed, stamina_changed, flask_changed,
    died.
  - camera.pull_back(distance, in_time, source) — boss big-move hook.
  - game_manager: victory_reached() — the HAND-OFF to the future
    transition sequence (out of scope here; the stub fires the signal).
  - AttackData normalized-window → AnimationPlayer call-track contract
    (documented Phase 11) — the seam future real animations plug into.

## 8. TESTING STRATEGY

  Unit / headless (run via `godot --headless --script` or GUT if
  adopted):
    - Combat core: single-hit, no double-hit, hitstop restore,
      i-frame skip (Phase 1).
    - Moveset tree: seeded chain sequences, overshoot→BACK-TO-PLAYER,
      chain-length variance 2..6 (Phase 4).
    - AI: initiator distribution vs §6 weights at 3/7/12m; buff
      thresholds (Phase 5).
    - Lightning: active only <50% HP; Dragon's Charge one-time
      (Phase 9).
  Integration (automated smoke, Phase 10):
    - Scripted player i-frame-rolls (0 dmg) then poise-breaks a
      scripted boss; death + victory end states fire.
  Manual QA checklist (the FEEL pass — human, Aaron):
    [ ] Roll feels like a mid-roll; i-frames land dodges into swings.
    [ ] Stamina pressure is real; can't spam.
    [ ] Light/heavy weight + hitstop feel ER-faithful.
    [ ] Boss pressures constantly, never idle >1.5s (except Pause).
    [ ] The same opener produces different-length chains (unmemorizable).
    [ ] Overshoots become attacks (no free punish on a turned back).
    [ ] The Pause creates genuine tension; punishes both attack and wait.
    [ ] Stagger window is a satisfying, tight punish.
    [ ] Below 50%: lightning integrates, doesn't become a separate mode.
    [ ] Dragon's Charge reads as "run, don't roll" and happens once.
    [ ] Void + boss-as-light-source looks like Elden Ring.
    [ ] "YOU DIED" is pixel-faithful (Cinzel, #C8986E).
